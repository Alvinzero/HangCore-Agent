# 航顺 AI 智能体项目任务进度清单

> 最后更新：2026-07-03
> 当前版本：`v0.1.3`
> 当前阶段：`nomiFun 底座 + Windows 发布链 + Kun Agent MVP 接入 + Skills 中文友好`
> 主仓库：`git@github.com:Alvinzero/HangCore-Agent.git`

## 1. 使用规则

- 当用户发送“查看任务清单”时，先读取本文件，再汇报当前进度、已完成任务、未完成任务、下一版本目标和主要风险。
- 每完成一个模块，必须先更新本清单，再提交到 GitHub，并发布一个新版本。
- 每个已完成模块都要记录版本号、关键提交、验证命令、GitHub Actions 结果和 Release 资产。
- Windows x64 是当前唯一发布目标；macOS / Linux 暂不纳入本阶段。
- Kun 不复制运行时代码进 nomiFun core，只通过 HangCore 自有 `kun-acp-adapter` 作为可选 Local Agent 接入。
- 旧 AionUi、旧 Kun、spec-kit 源码包、旧工具链源码包和构建残留不得进入安装包，除非某个阶段明确需要并重新评估边界。

## 2. 当前总览

当前项目已经完成从旧 `AionUi + Kun + spec` 到 `nomiFun + Kun Agent + Spec Kit` 的底座切换、Windows 首版发布链、Kun Agent 注册占位、`kun-acp-adapter` MVP，以及内置 Skills 在中文环境下的中文友好展示和搜索。

下一步重点不是继续做外壳，而是让 Kun Agent 对话链路更接近真实可用：approval / user input / tool call UI 映射、Kun runtime 自动发现或启动引导、Windows 真机安装与更新验收。

## 8. 风险与阻塞

- Release notes 自动生成仍有弱点：`v0.1.2` 的资产和 updater URL 正确，但 Release body / `latest.json.notes` 可能仍偏向 CI commit 标题。
- 本机 macOS 环境历史上缺少 `cargo` / `rustc`，Rust 验证主要依赖 GitHub Actions；后续本地 Rust 工具链应补齐。
- `kun-acp-adapter` MVP 依赖本机 Kun runtime 已启动；当前还不负责安装或自动拉起 Kun runtime。
- Kun approval / user input 是体验关键点，如果只映射文本流，会让 Kun Agent 看起来能聊但不能完整执行 coding loop。
- Windows 真机验收尚未闭环，不能只依赖 CI 产物判断用户可安装体验。
- 未来接入 HK64S8x 工具链时，必须把编译、烧录、文件写入等高风险动作纳入审批和审计。

## 9. 查看任务清单汇报格式

当用户发送“查看任务清单”时，按下面结构汇报：

1. 当前版本与阶段。
2. 总体进度判断：已完成哪些里程碑，当前卡在哪个里程碑。
3. 最近已完成模块：版本、模块名、核心变化、Release 状态。
4. 当前未完成重点：最多列 5 到 8 条最高优先级。
5. 下一步建议：明确下一版本号和要做的模块。
6. 阻塞或风险：只列会影响推进的真实问题。
