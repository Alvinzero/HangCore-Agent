## 1. 使用规则

- 当用户发送“查看任务清单”时，先读取本文件，再汇报当前进度、已完成任务、未完成任务、下一版本目标和主要风险。
- 每完成一个模块，必须先更新本清单，再提交到 GitHub，并发布一个新版本。
- 每个已完成模块都要记录版本号、关键提交、验证命令、GitHub Actions 结果和 Release 资产。
- Windows x64 是当前唯一发布目标；macOS / Linux 暂不纳入本阶段。
- Kun 不复制运行时代码进 nomiFun core，只通过 HangCore 自有 `kun-acp-adapter` 作为可选 Local Agent 接入。
- 旧 AionUi、旧 Kun、spec-kit 源码包、旧工具链源码包和构建残留不得进入安装包，除非某个阶段明确需要并重新评估边界。

## 2. 当前总览

当前项目已经完成从旧 `AionUi + Kun + spec` 到 `nomiFun + Kun Agent + Spec Kit` 的底座切换、Windows 首版发布链、Kun Agent 注册占位、`kun-acp-adapter` MVP、内置 Skills 在中文环境下的中文友好展示和搜索，以及 Kun Agent 复用系统模型服务商配置的兼容修复。

下一步重点从 M3 adapter 核心闭环转入 M4：`v0.1.5` 已发布，HangCore 选择 Kun Agent 后可通过 `kun-acp-adapter` 使用 Kun 原 HTTP/SSE runtime 和 AgentLoop；已补齐 ACP permission、Kun approval/user-input 回填、tool call lifecycle 映射和 adapter 设计文档。剩余 M3 工作是 Windows 真机安装、自动更新和真实 Kun runtime 长链路验收，以及专用 user-input 表单 UI。

## 8. 风险与阻塞

- Release notes 自动生成仍有弱点：`v0.1.4` 已由用户手动发布，但 Release body、`latest.json.notes`、Windows updater notes 和资产完整性仍需补验记录。
- 本机 macOS 环境历史上缺少 `cargo` / `rustc`，Rust 验证主要依赖 GitHub Actions；后续本地 Rust 工具链应补齐。
- `v0.1.5` 修复的是 adapter 对真实 Kun runtime 的核心协议闭环，但 Windows 真机安装、updater 资产和长链路人工验收仍不能省略。
- `kun-acp-adapter` 仍依赖本机 Kun runtime 或 `kun` CLI 可用；当前还不完整负责安装、跨路径自动发现或 Local Agents 状态可视化。
- Kun user-input 专用表单 UI 尚未完成；当前通过权限卡承载单题选项输入，多题或自由文本会保守取消。
- Windows 真机验收尚未闭环，不能只依赖 CI 产物判断用户可安装体验。
- 未来接入 HK64S8x 工具链时，必须把编译、烧录、文件写入等高风险动作纳入审批和审计。

## 9. 查看任务清单汇报格式

当用户发送“查看任务清单”时，按下面结构汇报：

1. 当前版本与阶段。
2. 总体进度判断：已完成哪些里程碑，当前卡在哪个里程碑。
3. 最近已完成模块：版本、模块名、核心变化、Release 状态。
4. 当前未完成重点：最多列 5 到 8 条最高优先级。
5. 下一步建议：明确下一版本号和要做的模块。
6. 阻塞或风险：只列会影响推进的真实问题。
