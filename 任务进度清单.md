## 1. 使用规则

- 当用户发送“查看任务清单”时，先读取本文件，再汇报当前进度、已完成任务、未完成任务、下一版本目标和主要风险。
- 每完成一个模块，必须先更新本清单，再提交到 GitHub，并发布一个新版本。
- 每个已完成模块都要记录版本号、关键提交、验证命令、GitHub Actions 结果和 Release 资产。
- Windows x64 是当前唯一发布目标；macOS / Linux 暂不纳入本阶段。
- Kun 不复制运行时代码进 nomiFun core，只通过 HangCore 自有 `kun-acp-adapter` 作为可选 Local Agent 接入。
- 旧 AionUi、spec-kit 源码包、旧工具链源码包和构建残留不得进入安装包；Kun runtime 只能作为明确的受管运行时资源进入安装包，不得混入 nomiFun core。

## 2. 当前总览

当前项目已经完成从旧 `AionUi + Kun + spec` 到 `nomiFun + 8位MCU Profile + Spec Kit` 的底座切换、Windows 首版发布链、8位MCU Profile 注册占位、`kun-acp-adapter` MVP、内置 Skills 在中文环境下的中文友好展示和搜索，以及 8位MCU Profile 复用系统模型服务商配置的兼容修复。

M3 纠偏已随 `v0.1.6` 发布：8位MCU Profile 默认不再绕过真实 runtime 直连模型服务商；provider fallback 只保留为 `KUN_PROVIDER_FALLBACK=1` 的显式诊断模式；adapter 可自动发现同级 Kun 源码并启动真实 Kun runtime，同时复用系统设置注入的模型、API Key 与 Base URL；SSE 解析已兼容 CRLF frame 边界，减少 Windows / Node 流式输出被缓冲到连接结束才显示的风险。

`v0.1.8` 已完成后台托管 Kun runtime 与 Markdown / Mermaid 渲染修复。`v0.1.9` 发布目标是把用户可见的旧 Kun-backed Agent 名称收敛为 `8位MCU Profile`，底层继续保留 `backend = "kun"`、`agent_builtin_kun`、`kun-acp-adapter` 和真实 Kun runtime loop。下一步继续 M4 CodingTask + SpecArtifact，同时补验干净 Windows 真机安装和自动更新端到端。

- Release notes 自动生成仍有弱点：`v0.1.4` 已由用户手动发布，但 Release body、`latest.json.notes`、Windows updater notes 和资产完整性仍需补验记录；`v0.1.7` 的 `latest.json.notes` 当前仍是 tag / commit 简句，后续需要强化为面向用户的更新说明。
- 本机 macOS 环境历史上缺少 `cargo` / `rustc`，Rust 验证主要依赖 GitHub Actions；后续本地 Rust 工具链应补齐。
- `v0.1.7` 已完成 GitHub Actions Windows Release 出包，仍需干净 Windows 真机验收。
- 受管 Kun runtime 已增加 Windows 安装包体积和 CI 打包时间；`v0.1.7` exe 约 148 MB，installer step 约 37 分钟，后续需优化 NSIS 压缩策略或 managed runtime 体积。
- `kun-acp-adapter` 仍需要更好的 Local Agents 状态可视化；即使受管 runtime 内嵌，资源损坏、杀毒隔离、端口占用和 token 配置错误仍需更明确提示。
- Kun user-input 专用表单 UI 尚未完成；当前通过权限卡承载单题选项输入，多题或自由文本会保守取消。
- Windows 真机验收尚未闭环，不能只依赖 CI 产物判断用户可安装体验。
- 未来接入 HK64S8x 工具链时，必须把编译、烧录、文件写入等高风险动作纳入审批和审计。

## 9. 查看任务清单汇报格式

当用户发送“查看任务清单”时，按下面结构汇报：

1. 当前版本与阶段。
2. 总体进度判断：已完成哪些里程碑，当前卡在哪个里程碑。
3. 最近已完成模块：版本、模块名、核心变化、Release 状态。
4. 当前未完成重点：最多列 5 到 8 条最高优先级。
5. 下一步建议：明确下一版本号和要做的模块。
6. 阻塞或风险：只列会影响推进的真实问题。
