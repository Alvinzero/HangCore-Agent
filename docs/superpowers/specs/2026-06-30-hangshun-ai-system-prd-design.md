## 2. 背景与目标

航顺内部存在两类高频智能化诉求：

1. 办公侧：员工需要用 AI 完成知识问答、文档生成、表格处理、PPT 初稿、定时任务等日常工作，但企业需要对智能体模板、知识库、工具权限、模型密钥和审计进行统一管控。
2. 研发侧：编程部门希望通过自然语言描述 MCU 研发需求，由 AI 在公司 Spec、芯片规则、工程模板和编码规范约束下生成可编译、可烧录的工程代码，并通过编译日志反馈持续修正。

航顺 AI 系统首发目标不是做泛化 SaaS，也不是做一个普通聊天客户端，而是做一个内部可落地的企业智能体桌面工作台。它以 AionUi 为桌面底座，办公侧复用 AionUi 的办公智能体能力，Coding 侧接入 Kun 风格 Agent Loop，并用 Spec-Kit 思路约束需求、规范、计划、任务和验证边界。公司知识库通过 Dify 或 MaxKB API 接入。

## 3. 产品定位

### 3.1 核心定位

航顺 AI 系统是面向航顺内部员工和编程部门的综合智能体桌面应用，提供 `办公` 与 `Coding` 两个主工作台。

- `办公`：面向日常办公智能体、知识问答、文档产物和自动化任务。
- `Coding`：面向 MCU 编程部门，通过自然语言驱动 8 位汇编工程和 32 位 C/Keil 工程的代码生成、编译、烧录确认和结果记录。

### 3.2 首发主闭环

MVP 首发闭环为：

`桌面端自然语言输入 -> AI 澄清需求 -> Spec/规则约束 -> 生成或修改工程文件 -> 自动编译 -> 日志解析与自动修复 -> 人工确认烧录 -> 查看硬件效果 -> 留痕审计`

办公侧闭环为：

`选择部门办公智能体 -> 输入办公任务或知识问题 -> 调用受控工具/知识库 -> 生成结果 -> 展示引用 -> 留痕审计`

## 4. 范围与非目标

### 4.1 MVP 范围

- AionUi 桌面底座改造，形成 `办公 / Coding` 双工作台。
- 办公智能体模板由部门管理员配置和发布。
- Dify / MaxKB 作为外部 RAG 服务接入。
- Coding 模块接入 Kun 风格 Agent Loop。
- Coding 模块使用 Spec-Kit 风格产物约束需求、规范、计划、任务和验收。
- 8 位 MCU 汇编 Pipeline：生成/修改汇编项目文件、自动编译、日志修复、人工确认烧录。
- 32 位 MCU C/Keil Pipeline：生成/修改 C 工程文件、调用 Keil CLI 编译、展示和归档日志，烧录能力按工具链成熟度以确认动作接入。
- 基础用户、部门、角色、工具权限、密钥保护和审计日志。

### 4.2 非目标

- 不做对外通用 SaaS 商业平台首发。
- 不做完整 OA、ERP、PLM 或 IDE 替代品。
- 不自研 RAG 底座、向量数据库、文档切分和知识库后台。
- 不首发移动端。
- 不支持 64 位 MCU。
- 不允许员工自由创建不受控的高权限个人智能体。
- 不允许 AI 直接自由执行任意 shell 命令。

## 5. 用户角色

| 角色 | 说明 | 主要能力 |
| --- | --- | --- |
| 普通员工 | 使用办公智能体处理日常任务 | 使用已发布办公智能体、知识问答、查看本人任务 |
| 编程人员 | MCU 研发人员 | 创建 Coding 任务、生成代码、编译、查看 Diff、人工确认烧录 |
| 部门管理员 | 部门智能体和知识绑定负责人 | 配置本部门智能体模板、知识库绑定、成员可用能力 |
| 系统管理员 | 系统级配置负责人 | 管理模型、密钥、工具链、安全策略、审计 |
| 审计员 | 合规和过程留痕查看者 | 只读查看日志、审批记录、工具调用记录 |

## 6. 产品原则

- `内部试点优先`：先服务航顺内部真实办公和 MCU 编程场景。
- `双工作台清晰分离`：办公和 Coding 的工具、流程、权限和产物分开管理。
- `Spec 约束优先`：Coding 输出必须受公司规范、芯片规则、工程模板和验收标准约束。
- `半自动执行`：AI 可自动读文件、改代码、编译和修复；烧录和高风险动作必须人工确认。
- `工具白名单`：Agent 只能调用受控工具，不能自由拼接系统命令。
- `全链路留痕`：智能体调用、知识库引用、代码变更、编译、烧录、审批都必须可追溯。
- `外部能力可替换`：RAG、模型、工具链适配层应保持可替换，不强耦合单一厂商。

## 7. 桌面端产品结构

### 7.1 单壳双工作台

桌面端采用类似 Claude Code / Cowork 的左侧主导航形态，顶部提供 `办公` 与 `Coding` 两个主模式切换。

`办公` 模式侧边栏包含：

- 新任务
- 办公项目
- 产物
- 定时任务
- 智能体定制
- 最近记录

`Coding` 模式侧边栏包含：

- 新建研发任务
- 项目/仓库
- Spec
- Plan
- Tasks
- 工具链
- 构建记录
- 烧录记录
- 最近会话

### 7.2 主工作区

办公模式主区域为办公智能体入口，用户选择部门发布的受控智能体后输入任务。

Coding 模式主区域采用 `聊天澄清 + 结构化产物并行` 布局：

- 对话区：自然语言需求、AI 澄清、执行反馈。
- 结构化区：需求单、Spec、Plan、Tasks、Diff、编译日志、烧录确认。
- 状态区：当前 Pipeline、编译状态、烧录状态、工具调用记录、错误摘要。

## 8. 总体架构与职责边界

### 8.1 AionUi

AionUi 作为桌面应用底座，负责：

- Electron 桌面壳。
- 主界面和导航。
- 办公智能体体验。
- 模型配置和基础会话。
- MCP 管理。
- 文档类能力。
- 定时任务入口。

### 8.2 Kun Runtime

Kun Runtime 作为 Coding Runtime 参考和改造来源，负责：

- Agent Loop。
- 事件流输出。
- 工具调用流程。
- 多轮编译日志修复。
- 审批和人工输入 gate。
- Coding 任务状态推进。

Kun Runtime 不直接与 AionUi UI 混写，应通过 HTTP/SSE/IPC 等稳定边界接入。

### 8.3 Spec-Kit

Spec-Kit 作为规范驱动层，负责提供流程和产物约束：

- 需求澄清。
- Spec。
- Plan。
- Tasks。
- Checklist。
- 验收标准。

在本产品中，Spec-Kit 不需要作为独立产品入口暴露，而应内嵌在 Coding 流程里，约束 AI 代码生成与验证。

### 8.4 Toolchain Adapter

Toolchain Adapter 负责封装 MCU 工具链：

- 8 位 MCU 编译 CLI。
- 8 位 MCU 烧录 CLI。
- 32 位 Keil CLI。
- 日志解析。
- 产物定位。
- 命令白名单。
- 风险等级与审批策略。

Agent 只能通过 Toolchain Adapter 调用工具，不能直接执行任意 shell 命令。

### 8.5 Dify / MaxKB

Dify / MaxKB 作为外部 RAG 服务，航顺 AI 只做 API 接入：

- 知识库选择。
- 问答调用。
- 引用来源展示。
- 部门权限映射。
- 结果注入 Agent 上下文。
- 调用审计。

文档上传、切分、向量化、知识库维护和底层模型配置仍在 Dify / MaxKB 后台完成。

## 9. 办公模块需求

### 9.1 受控办公智能体模板

办公板块采用部门级智能体模板机制。

系统管理员配置全局可用模型、工具、密钥和安全策略。部门管理员发布本部门智能体模板。普通员工只能使用已发布模板。

模板字段包括：

- 名称
- 说明
- 标识
- System Prompt
- 默认模型
- 可用工具
- 绑定知识库
- 输出格式
- 审批规则
- 版本号
- 发布状态

### 9.2 办公智能体能力

MVP 支持：

- 公司知识问答。
- 文档生成和改写。
- 表格处理。
- PPT 初稿。
- 定时任务。
- 基础办公任务记录。

### 9.3 办公审计

每次办公智能体调用记录：

- 用户。
- 部门。
- 智能体模板。
- 模板版本。
- 输入摘要。
- 输出摘要。
- 工具调用。
- 知识引用。
- 失败原因。
- 时间戳。

## 10. Coding 模块需求

### 10.1 Coding 模块定位

Coding 模块面向编程部门人员，目标是通过自然语言驱动 MCU 工程代码生成与验证。

核心闭环为：

`自然语言需求 -> AI 澄清 -> Spec 约束 -> 生成项目文件 -> 自动编译 -> 日志修复 -> 编译通过 -> 人工烧录 -> 查看效果 -> 留痕`

Coding 模块不是通用代码聊天工具，而是受公司规范、芯片资料、项目模板和工具链约束的工程代码工作台。

### 10.2 双 Pipeline 架构

8 位 MCU 与 32 位 MCU 差异显著，必须设计为两个独立 Pipeline。

两者共享：

- 自然语言需求入口。
- Spec-Kit 风格规范约束。
- Kun 风格 Agent Loop。
- Diff 展示。
- 编译日志展示。
- 审计记录。
- 人工烧录确认。
- 最大循环次数和超时。

两者独立：

- 代码语言。
- 代码生成策略。
- 工程模板。
- 编译器。
- 烧录工具。
- 日志解析规则。
- 验收标准。

### 10.3 8 位 MCU Pipeline

8 位 MCU Pipeline 面向汇编语言项目。

输入：

- 自然语言需求。
- 目标芯片型号。
- 公司 8 位 MCU Spec。
- 指令规则。
- 寄存器规则。
- 汇编语法规则。
- 项目模板。
- 示例代码。

输出：

- 汇编源码。
- 工程配置。
- 可编译项目目录。
- 编译产物。
- 编译日志。
- 烧录结果。

流程：

1. 用户用自然语言描述需求。
2. AI 根据 8 位 MCU Spec 和汇编规则澄清需求。
3. AI 生成或修改汇编项目文件。
4. 系统调用 8 位 MCU 编译 CLI。
5. 编译失败时，Agent 读取日志并修复汇编代码。
6. 达到编译通过或达到最大修复次数后停止。
7. 编译通过后，用户人工点击烧录。
8. 系统调用 8 位 MCU 烧录 CLI。
9. 系统记录烧录日志和结果。

验收：

- 生成的汇编工程能被 8 位编译 CLI 编译。
- 编译失败时至少完成一轮日志解析和修复。
- 烧录前必须人工确认。
- 烧录结果可查看和归档。

### 10.4 32 位 MCU Pipeline

32 位 MCU Pipeline 面向 C 语言和 Keil 工程。

输入：

- 自然语言需求。
- 目标芯片型号。
- 公司 32 位 MCU 规范。
- C 编码规范。
- Keil 工程模板。
- SDK 和芯片库。
- 启动文件和工程配置。
- 示例代码。

输出：

- C 源码。
- 头文件。
- Keil 工程文件或可被 Keil CLI 编译的项目目录。
- 编译产物。
- 编译日志。
- 烧录动作记录。

流程：

1. 用户用自然语言描述需求。
2. AI 根据 32 位 MCU 规范、C 规范、Keil 工程规则澄清需求。
3. AI 生成或修改 C 工程文件。
4. 系统调用 Keil CLI 编译。
5. 编译失败时，Agent 读取日志并修复 C 代码或工程配置。
6. 编译日志展示并归档。
7. 烧录能力作为人工确认工具接入，按具体 32 位工具链成熟度逐步完善。

验收：

- 生成或修改的 C 工程能通过 Keil CLI 编译。
- 编译日志可展示、摘要和归档。
- 烧录动作不自动执行。

### 10.5 半自动权限策略

默认权限模式为半自动：

- AI 可自动读取项目内文件、编译配置、芯片资料、历史日志和知识库内容。
- AI 可生成和修改代码，但每次改动必须形成 Diff。
- AI 可自动调用白名单编译命令。
- AI 可根据编译错误自动进入下一轮修复。
- 烧录动作必须人工确认。
- 删除文件、批量移动文件、修改工具链配置、执行非白名单命令必须人工确认。
- 每次 Agent 决策、工具调用、代码 Diff、编译日志、烧录结果都进入审计记录。

## 11. Toolchain Profile

MVP 内置航顺现有工具链配置，但数据模型按可配置 Profile 设计。

字段包括：

- `profileId`
- `name`
- `targetArch`: `8bit` 或 `32bit`
- `language`: `assembly` 或 `c`
- `mcuFamily`
- `projectTemplate`
- `compileCommandTemplate`
- `flashCommandTemplate`
- `workingDirectory`
- `env`
- `artifactPatterns`
- `logParser`
- `riskLevel`
- `requiresApproval`
- `enabled`

MVP 阶段不要求管理员可视化创建 Profile，但实现上不能把命令散落到 Agent Prompt 中。

## 12. RAG 知识库需求

MVP 中，Dify / MaxKB 只作为外部知识库服务。

桌面端负责：

- 保存 Dify / MaxKB API 配置。
- 维护部门到知识库的映射。
- 在办公智能体和 Coding 智能体中选择可用知识库。
- 调用知识库问答接口。
- 展示引用来源。
- 将检索结果注入 Agent 上下文。
- 记录知识库调用审计。

Dify / MaxKB 后台负责：

- 文档上传。
- 文档切分。
- 向量化。
- 检索配置。
- 知识库维护。
- 底层模型配置。

## 13. 数据模型

### 13.1 组织与用户

- `User`：员工账号、姓名、部门、角色。
- `Department`：部门工作区。
- `Role`：系统管理员、部门管理员、编程人员、普通员工、审计员。

### 13.2 办公侧

- `AssistantTemplate`：部门智能体模板。
- `AssistantTemplateVersion`：模板版本。
- `OfficeTask`：办公任务或问答任务。
- `KnowledgeBinding`：部门/智能体与 Dify/MaxKB 知识库绑定。

### 13.3 Coding 侧

- `CodingProject`：研发项目，绑定 8 位或 32 位 Pipeline。
- `CodingTask`：自然语言需求进入后的研发任务。
- `SpecArtifact`：Spec、Plan、Tasks、验收标准等规范产物。
- `CodeChange`：AI 生成或修改的代码变更。
- `ToolchainProfile`：工具链配置。
- `BuildRun`：一次编译执行记录。
- `FlashRun`：一次烧录执行记录。

### 13.4 治理侧

- `Approval`：高风险动作确认。
- `Secret`：模型 Key、Dify/MaxKB Key、工具链凭据等敏感配置。
- `AuditLog`：关键操作留痕。

## 14. 权限边界

| 角色 | 权限 |
| --- | --- |
| 系统管理员 | 管理全局模型、密钥、工具链、安全策略、审计 |
| 部门管理员 | 管理本部门智能体模板、知识库绑定、部门成员能力 |
| 编程人员 | 创建 Coding 任务、生成代码、编译、查看 Diff、人工确认烧录 |
| 普通员工 | 使用办公智能体、提交自然语言需求、查看本人任务 |
| 审计员 | 只读查看日志、审批记录、工具调用记录 |

关键约束：

- 普通员工不能修改智能体模板、工具链 Profile、密钥配置。
- AI Agent 不能绕过权限直接调用系统命令。
- 所有编译/烧录都必须关联用户、项目、工具链 Profile 和日志。
- 所有知识库访问必须通过部门绑定关系校验。
- 所有 CodeChange 必须可追溯到自然语言需求和 SpecArtifact。

## 15. MVP 验收标准

### 15.1 桌面工作台

- 应用启动后，左侧可在 `办公 / Coding` 两个模式之间切换。
- 两个模式各自有独立导航、最近任务和主输入区。
- 用户身份、部门上下文、模型配置、知识库配置在两个模式中保持一致。

### 15.2 办公

- 部门管理员能发布至少 1 个办公智能体模板。
- 普通员工能使用该模板完成一次文档或知识问答类任务。
- 办公智能体能调用 Dify/MaxKB 外部知识库 API。
- 回答结果能展示引用来源。
- 调用过程能形成审计记录。

### 15.3 Coding 8 位 MCU

- 编程人员能用自然语言提出 1 个 8 位 MCU 编程需求。
- AI 能基于公司 8 位 MCU Spec 和汇编规范生成或修改汇编项目文件。
- 系统能自动调用 8 位编译 CLI。
- 编译失败时，Agent 能读取日志并至少完成 1 轮自动修复。
- 编译通过后，用户能手动点击烧录。
- 系统能记录烧录结果和日志。

### 15.4 Coding 32 位 MCU

- 编程人员能用自然语言提出 1 个 32 位 MCU C 语言需求。
- AI 能基于公司 32 位 MCU / Keil 工程规范生成或修改 C 工程文件。
- 系统能调用 Keil CLI 完成编译。
- 编译日志能展示并归档。
- 烧录动作作为可确认工具接入，按工具链成熟度逐步完善。

### 15.5 安全与审计

- AI 不能自由执行非白名单 shell 命令。
- 烧录动作必须人工确认。
- 文件变更必须能查看 Diff。
- 编译、烧录、知识库调用、智能体调用都要有日志记录。
- 普通员工不能修改部门智能体模板和工具链配置。

## 16. 里程碑

### 阶段 0：源码评估与底座确认

- 确认 AionUi 可改造范围。
- 梳理 AionUi 办公智能体、模型配置、MCP、文档能力、侧边栏导航。
- 梳理 Kun Agent Loop、HTTP/SSE 事件、工具调用、审批机制。
- 梳理 Spec-Kit 的 Spec、Plan、Tasks、Checklist 产物格式。
- 输出技术复用清单：直接复用、改造复用、只参考、不采用。

### 阶段 1：桌面双工作台

- 改造首页和左侧导航。
- 形成 `办公 / Coding` 双模式。
- 建立部门上下文、最近任务、项目列表、主输入区。

### 阶段 2：办公智能体 MVP

- 接入 AionUi 现有办公智能体能力。
- 增加部门模板、模板版本、知识库绑定、工具权限。
- 接入 Dify/MaxKB API。
- 完成一次知识问答与引用展示。
- 完成办公调用审计。

### 阶段 3：Coding 8 位 MCU 主链路

- 搭建 Coding Runtime 边界。
- 建立 8 位 MCU 汇编 Pipeline。
- 接入公司 8 位 Spec、指令、寄存器、工程模板约束。
- 调用已有 8 位编译 CLI。
- 解析日志并驱动 Agent Loop 修复。
- 编译通过后人工确认烧录。

### 阶段 4：Coding 32 位 MCU 辅链路

- 建立 32 位 C/Keil Pipeline。
- 接入 Keil 工程模板、C 编码规范、SDK/芯片库约束。
- 调用 Keil CLI 完成编译。
- 烧录作为确认动作预留或按工具链成熟度接入。

### 阶段 5：治理与审计增强

- 完善角色权限。
- 完善工具调用白名单。
- 完善敏感配置保护。
- 完善编译、烧录、知识库、智能体调用审计。
- 增加失败回滚、日志导出和演示数据。

## 18. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| AionUi 与 Kun 架构耦合过深 | 改造复杂、维护困难 | 使用清晰 Runtime 边界，Coding 能力通过 HTTP/SSE/IPC 接入 |
| Kun 许可证限制 | 未来商用或对外交付存在合规风险 | 内部试点先做技术验证，商业化前进行许可证审查 |
| 8 位汇编生成质量不稳定 | 编译失败或硬件行为不可控 | 强制引入指令规则、寄存器规则、模板工程、编译日志闭环 |
| 32 位 Keil 工程结构复杂 | AI 容易误改工程配置 | 以工程模板和受控文件清单约束生成范围 |
| 烧录存在硬件风险 | 误烧录或设备状态不一致 | 烧录必须人工确认，展示芯片、设备、产物和命令摘要 |
| RAG 质量受外部系统影响 | 回答不稳定或引用不准 | MVP 只做 API 接入和引用展示，知识库治理在 Dify/MaxKB 后台完成 |
| 密钥和工具链配置泄露 | 安全风险 | Secret 集中管理，普通用户不可见，日志脱敏 |

## 19. 后续版本方向

- 外部入口：企业微信、飞书、钉钉、OA、客户反馈系统。
- 更多办公自动化流程。
- 知识库管理入口逐步内嵌。
- 更多 MCU 工具链 Profile。
- 更完整审批流。
- 研发任务看板。
- 与代码仓库、CI、制品库集成。

- 对外商业化多租户能力。

## 20. 结论

航顺 AI 系统 V1.0 应聚焦内部可落地的双工作台：办公侧用受控智能体提升日常办公效率，Coding 侧围绕 8 位汇编和 32 位 C/Keil 两条 MCU Pipeline 打通自然语言到可编译、可烧录工程文件的闭环。

产品成功的关键不是做一个万能聊天框，而是把 AionUi 的桌面和办公能力、Kun 的 Agent Loop、Spec-Kit 的规范驱动、Dify/MaxKB 的知识库能力，以及航顺已有 MCU 编译/烧录工具链，组合成一个可控、可审计、可扩展的企业工程智能体工作台。
