# 全局 Agent 工作区交付与中文输出设计

日期：2026-07-16

## 背景

当前会话右侧的“临时空间/项目”面板展示真实工作区文件。部分 Agent 已经在工作区创建文件，但文件树只监听了部分单次工具事件，遗漏组合工具事件和会话结束后的最终校准，因此可能需要手动刷新才能看到新文件。

另一些 Agent 只把代码或资料作为聊天 Markdown 返回，并没有创建真实文件。此时右侧文件树没有可展示的文件。8位MCU Profile 的 Kun 提示合同还主要使用英文，也没有全局中文输出约束，导致用户可见的思考和代码说明出现英文。

## 目标

1. 当前所有会话 Agent 和未来新增 Agent 都自动遵守同一交付规则，不按 Agent 名称或后端类型维护白名单。
2. 用户要求生成代码、文档或资料交付物时，Agent 必须把最终交付物写成当前工作区内的真实文件，不能只返回聊天代码块。
3. 所有用户可见的自然语言，包括公开展示的思考、进度、工具摘要、最终说明和代码注释，默认使用简体中文。
4. 编程语言关键字、汇编指令、寄存器名、API 名、库名、命令、路径和必要标识符保持原样，不能为了中文化破坏可执行性或可编译性。
5. 文件一旦创建或修改，右侧工作区在会话进行中尽快刷新，并在任意 Agent 结束本轮后强制完成一次最终刷新。

## 非目标

- 不从普通聊天 Markdown 中猜测文件名并自动拆出代码块。
- 不为没有真实文件的回答制造虚拟文件树节点。
- 不改变 Kun 原生 runtime loop、交互式 `user_input`、流式事件或工具执行能力。
- 不把内部不可见推理暴露给用户；中文约束只作用于产品实际展示的思考/进度内容。
- 不在本轮重构工作区文件树、预览器或 Agent 协议本身。

## 方案比较

### 方案 A：只增强前端刷新

优点是改动小，能够让已经落盘的文件更快出现。缺点是 Agent 只返回代码块时仍没有真实文件，无法满足统一交付要求。

### 方案 B：分别修改每个 Agent 的提示词

可以让现有 Agent 主动写文件，但 Nomi、ACP、OpenClaw、Nanobot、Remote 和后续 Agent 都要重复接线，容易漏掉自定义或新注册 Agent，长期会发生规则漂移。

### 方案 C：公共发送边界注入 + 公共事件层刷新（采用）

在 `AgentInstance::send_message` 的统一入口注入平台交付合同，所有当前 Agent 类型和未来新增枚举分支都会自动继承。前端不区分 Agent 类型，在统一 `message.stream` 上处理文件工具事件，并在 `finish`/`error` 时做最终刷新。该方案同时解决“没有落盘”和“落盘后未显示”两个问题。

## 架构设计

### 1. 全局 Agent 交付合同

新增一个纯函数模块，负责把平台合同注入 `SendMessageData.content`。`AgentInstance::send_message` 在调用具体 `IAgentTask` 前统一执行该函数。

合同内容要求：

- 用户要求生成代码、文档、配置、报告或资料时，把最终交付物写入 `workspace()` 指向的当前工作区。
- 优先使用用户指定的路径和文件名；用户未指定时使用清晰、稳定且与任务相关的名称。
- 不得只在聊天中返回代码围栏而不落盘；最终回答同时说明真实文件路径。
- 文件写入失败时必须如实报告错误和目标路径，不得声称文件已经创建。
- 所有用户可见自然语言和代码注释使用简体中文，但保留源代码语法、汇编指令、寄存器、标识符、路径和命令原文。

合同使用稳定标记包裹，并实现幂等检查，避免某些内部重试或转发链重复注入。原始用户消息仍按当前逻辑持久化，合同只进入 Agent 执行输入，不显示在用户聊天气泡中。

### 2. 覆盖当前和未来 Agent

当前生产 Agent 类型为 ACP、Nomi、OpenClaw、Nanobot、Remote，它们都通过 `AgentInstance::send_message` 进入具体管理器。自定义 ACP Agent 也走同一入口。

以后新增 Agent 时必须新增 `AgentInstance` 枚举分支并实现 `IAgentTask`。由于公共注入位于枚举的统一转发方法，新类型不需要复制提示词即可继承合同。代码审查和测试将锁定这一不变量，禁止在具体 Agent 名称上建立白名单。

一个 Agent 若要作为完整的工作台 Agent 发布，必须把 `workspace()` 作为真实可写执行目录，或提供等价的工作区写入能力。无法访问该工作区的远程实现不能伪造交付成功，必须返回明确的不支持/写入失败信息。

### 3. 8位MCU Profile / Kun 规则

保留现有 Kun Markdown、Mermaid、原生 `user_input` 和 Agent loop 合同。全局合同负责中文和文件交付；Kun 专属合同继续负责 MCU 参数确认、交互卡片、工具调用和最终代码质检。

全局合同在每次进入 `AgentInstance::send_message` 时幂等注入，因此新会话、恢复的旧会话、模型故障转移和继续对话都会立即获得中文与文件交付要求，不需要清空历史会话。Kun 现有的恢复预设注入仍只负责其专属交互合同。

### 4. 工作区刷新策略

从 `ChatWorkspace` 中提取纯判断函数，统一处理 `message.stream`：

- `acp_tool_call`：编辑、执行或完成的非读取工具触发节流刷新。
- `tool_call`：完成的非团队/非纯读取工具触发节流刷新。
- `tool_group`：任一完成的可能写文件工具触发节流刷新。
- `finish`：不区分 Agent 类型，立即执行最终刷新，确保未知或未来事件协议仍能在本轮结束时显示真实文件。
- `error`：立即执行最终刷新，因为失败前可能已经产生部分有效文件。

实时刷新沿用现有 2 秒节流，避免工具密集调用造成文件树抖动。最终刷新会清理待执行定时器并立即加载一次，保证会话结束后右栏状态确定。

已有 `workspace-has-files` 事件继续负责在临时空间首次出现文件时展开右侧面板，不新增第二套展开状态。

## 数据流

1. 用户发送原始消息，ConversationService 按现有逻辑保存原始内容。
2. `AgentInstance::send_message` 为执行副本注入全局交付与中文合同。
3. 具体 Agent 在当前 `workspace()` 中创建或修改真实文件。
4. 工具事件通过统一 `message.stream` 到达前端，文件树进行节流刷新。
5. 任意 Agent 发出 `finish` 或 `error` 后，文件树执行最终刷新。
6. 发现文件后沿用现有工作区事件自动展开，用户可在右侧预览、选择和下载。

## 异常处理

- 工作区暂时不可读时保留现有树，不使用空响应覆盖已显示文件。
- 最终刷新失败只记录错误，不阻断聊天结果，也不删除旧树状态。
- Agent 写入失败时由 Agent 明确报告，不由前端生成虚假文件。
- 非文件工具、团队编排工具和纯读取工具不触发高频实时刷新；最终刷新仍提供兜底。
- 重试、模型故障转移和系统续跑使用幂等合同，确保每次实际发送最多注入一次。

## 测试设计

### Rust

- 全局合同会注入任意 `SendMessageData`，且原始用户内容保持在末尾。
- 已含合同标记的内容不会重复注入。
- 合同包含工作区真实文件交付、简体中文和“保留代码语法”的约束。
- `AgentInstance::send_message` 的公共路径应用合同，测试不依赖具体 Agent 名称。
- Kun 首次会话和恢复会话继续获得原生交互合同，并同时继承全局合同。

### TypeScript

- ACP 编辑/执行工具、普通完成工具和组合工具会触发实时刷新。
- 团队工具和纯读取工具不会触发多余实时刷新。
- 任意 `finish` 和 `error` 都会触发最终刷新，不依赖 `eventPrefix` 或 Agent 类型。
- 最终刷新会清理节流窗口并执行一次确定性加载。

### 回归验证

- 8位MCU Profile 生成 ASM：思考说明和注释为中文，指令/寄存器保持原样，ASM 文件出现在右侧临时空间。
- Nomi、内置 ACP、自定义 ACP 各生成一个代码或 Markdown 文件，均在本轮结束前后显示。
- OpenClaw、Nanobot、Remote 在具备工作区写入能力时遵守相同合同；不具备时明确失败，不显示虚假文件。
- 普通问答不强制创建无意义文件。
- 现有安装包优化和更新弹窗改动不纳入本功能提交。

## 验收标准

1. 实现代码中不存在按 `nomi`、`kun`、`codex` 等名称决定是否注入交付合同的白名单。
2. 新增 Agent 类型只要通过 `AgentInstance::send_message`，无需额外配置即可继承合同。
3. 用户要求生成代码或资料时，最终交付物是工作区真实文件，并能在右侧面板看到。
4. 任意 Agent 结束或报错后，右侧文件树与磁盘真实状态一致。
5. 用户可见思考、步骤、说明和代码注释为简体中文；代码语法与 MCU 指令不被翻译破坏。
